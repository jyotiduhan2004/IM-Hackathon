# V12 North Star — page-as-concept reframe

This document captures the v12 design conversation between Amit + Claude on
2026-04-18/19, intended to survive a context compaction. A post-compaction
Claude should read this doc first to pick up the thread without losing the
design intent.

## The reframe in one paragraph

The wiki's purpose is **institutional intuition transfer**, not documentation.
It should encode the hidden curriculum — the context that experienced IndiaMart
PMs and engineers carry in their heads but never write down. "Why does BL
recommendation have negative MCAT weights? Why 30-min polling instead of 5-min?
Why is Seller BL separate from Seller Dashboard?" These are the answers a new
joiner takes 18 months to acquire through osmosis. The wiki's job is to
compress that acquisition. If we succeed, we've built something genuinely new.
If we fail, we've built better-formatted email archives.

## Why v12 exists (what changed between v11 and here)

The Cycle 10 smoke 30 (run `6a2a89ae`, 2026-04-18) produced 7 content pages
at mean grade C+, validating the v11 stack mechanically (6 of 9 units PASS,
glob timeouts eliminated, `get_thread_context(concise) → read_file` chain
dropped 96.8% → 50%) — but the empirical archetype research the same afternoon
found:

- **0 of 303** existing topic pages match the v9-U1 canonical schema
  (Summary / Current state / Why it matters / Key decisions / Recent changes /
  Open questions / Related pages / References) at the 5-of-8 bar.
- **218 of 303 (72%)** converge on an organic design-doc shape the prompt
  never taught (Overview / Background / Implementation / Impact / Testing /
  Way Forward).

My first instinct was to codify the organic 7-slot shape and ship it as
`V12-U1 schema replacement`. Amit pushed back correctly: **that's the same
thread-mirror anti-pattern with nicer vocabulary**. Agent-writing-design-doc
is still agent-describing-the-email, not agent-writing-a-concept. The schema
isn't broken; the framing is.

v11-U7's reviewer was actually doing the right thing by passing coherent
alternative structures — we were measuring compliance to a target that
conflicted with the actual task framing.

## Design decisions (Q&A-derived)

### What IS a topic page?

**Concept + initiative (both), but concepts get priority.**

- A durable concept = a thing-in-the-world that will still exist in 6 months
  regardless of this specific work. *Seller BL API* ✓. *Lens 2.0* ✓.
  *ISQ scoring policy* ✓.
- An initiative = one-off project / launch / quarter's work.
  *Jan 2026 BL optimization* = an initiative. For now, **initiatives can
  just be raw email threads** — we don't need initiative pages yet. Build
  them up later if the data asks for it.
- Threshold for minting a concept page: **durability test**. If the subject
  will outlive this quarter's work, it's a concept.

**Vocabulary note (key):** Amit rejected the "experiment vs decision" binary —
"all pages are just pages; there's a superlink right there." Wikipedia model.
Don't mint new page_types (experiment, incident, case-study, learning). One
page type; content + tags + linking do the categorization work. URL directories
stay (`wiki/topics/`, `wiki/systems/`, `wiki/people/`) for continuity only, not
as a content-shape constraint.

### What does a page's `## Summary` reflect?

**Always current truth.** Not lineage. History lives in `## Recent changes`
and (when significant) linked supersession/experiment pages.

When info is contradicted by a new email (e.g., 30-min polling rolled back):
- Small changes → rewrite the contradicted sentence + add a dated bullet to
  `## Recent changes`.
- Significant changes → rewrite + mint a new experiment/supersession page
  linked from the topic.
- **Nothing vanishes. Strikethrough is offensive.** Old info moves into a
  collapsible archive (`<details>` tag works in mkdocs) or a quarterly
  summary section, but never gets struck through or deleted.
- Frame it as experiments / iteration, not "decisions we got wrong".

### Who is the primary reader?

**New joiner (primary)**, then future-you, then leadership, then on-call.

Amit's deeper framing: "Anyone can learn engineering or sales or finance. What
takes time is the INTUITION — about IndiaMart's customers, the way we work,
real-world hands-dirty experience. How do we squeeze the time to get to that
level?"

A page that works for a new joiner works for everyone else (depth comes from
linking + references, not re-writing). Design for the new joiner; everyone
else is downstream-satisfied.

**Invariant to hold: agent-friendly == new-joiner-friendly.** If a mechanism
(footnotes, backlinks, search) helps only one, it's probably wrong. Design
them symmetrically.

### How do we extract institutional intuition?

**Explicit 5W+IndiaMart expert questions baked into the prompt.**

Generic base:
- **WHAT**: which customer segment / product / API / page?
- **WHY**: what customer problem / business reason / historical constraint?
- **HOW**: which team/SBU owns it? which systems involved? what was tried
  before and why did it fail?
- **WHO**: stakeholders, leaders, customer groups, team names
- **WHEN**: timeline, state (shipped / experimental / archived)
- **WHERE**: surface (mobile / desktop / m-site / exports)

Per-domain flavor: `platform-reliability` topics answer tech-heavy questions
(scale, latency, failure modes). `growth-monetization` topics answer
business-heavy questions (segment, impact, conversion). `trust-safety` answers
threat-model questions. Same 5W skeleton, different emphasis.

### Raw email references — how accessible from a page?

**Inline Wikipedia-style footnotes** `[^msg-abc123]` attached to claims.

Why this over per-section sources: the agent has to own claim → source mapping
as it writes prose. That discipline IS the concept-reframe's enforcement
mechanism. Per-section sourcing lets the agent get away with "this section
cites these threads" at thread granularity — the filing-cabinet failure mode
in disguise.

Cost is moderate: agent writes `[^id]` markers; viewer already renders them
(markdown supports natively). Worth paying in v12.

### Attribution policy

**Decision-makers + experiment-owners only.**

`[[people/raghav-khurana]] recommended the 30-min interval based on Verification
rate limits.[^thread-abc]` — wikilink when the person is accountable (owner,
lead, approver). Don't wikilink every speaker. Avoid the v11-U8 auto-stub /
garbage-slug regression.

### Skip bar

**Tighten based on concept value, not length.** Current trivial-skip is
roughly length-based; new framing:

- Skip: OOO, calendar invites, pure acknowledgements with zero facts.
- Don't skip: one-liners like "rolled out to 10% more users" — that IS a
  `## Current state` update.

Expect trivial_skip rate to drop from ~50% → ~20% under v12.

## Success metric (how do we know v12 worked?)

Amit's framing: **not per-page grade, wiki-as-graph**. Can a reader navigate
Wikipedia-style — land on A, click to B, click to C, reach the raw email /
reference? The ENTIRE WIKI gets answers, not every page.

This combines two signals:
1. **New-joiner answerability** (LLM-judge proxy): simulated new joiner
   tries to answer expert-questions from a page; fails on the page but
   can they get there by following 1-2 wikilinks? Measures graph quality.
2. **Reader feedback** (future, post-v12): chatbot / search layer logs
   every question + whether a satisfying answer was found. Honest signal;
   slowest to accumulate.

Both deferred post-v12-tight. See § Scaling the measurement loop.

## V12 scope (tight → medium → large, sequenced)

### V12-U0 — pre-work: scale the measurement loop

Before any prompt work, build the scoring infrastructure so we stop
spot-checking 10 pages by hand.

**A. Deterministic scorer (ship first, continuous)**:
- `scripts/score_wiki.py` walks all pages, computes heuristic score 0-10:
  - Has `## Why it matters` with ≥2 sentences? (+2)
  - Body ≥500 words AND ≥3 wikilinks? (+2)
  - `source_threads:` has ≥2 (concept vs one-shot)? (+2)
  - No thread-subject H2s (`Launch Announcement`, `Bug report`, `Vote of
    thanks`)? (+2)
  - Mentions ≥1 people page by wikilink? (+1)
  - `last_compiled` within 30 days? (+1)
- Output CSV of slug → score + notes. Sort descending.
- Hook into Langfuse as a per-run score.
- Free, repeatable, runs on every compile.

**B. LLM-judge (one-time, ~$30 for 306 pages)**:
- "New-joiner simulator" prompt on each page.
- Rate 1-5 on WHAT / WHY / WHO-WHEN answerability. Generate 3 follow-up
  questions the page left unanswered.
- Re-run only on modified pages per compile cycle (~$2-5 ongoing).

**Outcome**: ranked list of 306 pages. Amit reviews top 10 best + bottom 20
worst + 10 random-middle = 40 pages (vs 306). Top 5 worst become v12's
"what NOT to do" few-shot examples; top 5 best become the golden anchors.

### V12-U1-U4 — tight scope (prompt reframe + footnotes + revision)

Ship these four together in a single /batch round, one PR each:

1. **U1 — Prompt reframe: "update concept page".**
   - Rewrite `scripts/compile_all.py` per-batch instruction from "Compile
     the following N emails into wiki pages" to "Update or create the
     CONCEPT page that best describes the subject of these emails. The
     emails are EVIDENCE — what was announced, built, tested, decided.
     The page is a CONCEPT — a durable description of the feature /
     initiative / decision / system itself, not a log of the thread."
   - Rewrite `src/compile/prompts.py` `<workflow>` + `<page_types>` to
     teach the concept-vs-thread distinction with concrete before/after
     examples (Summary line that reads as email-intro vs as concept-def).
   - Add a new `<concept_vs_thread>` section with explicit guidance.

2. **U2 — 5W+IndiaMart expert questions embedded in prompt.**
   - Prompt teaches the 5W skeleton + domain-specific flavor.
   - Per-domain examples: one platform, one growth, one trust-safety.
   - critique.py adds deterministic check: does the page mention
     stakeholder(s) / metric(s) / surface(s) / historical context? (Low
     bar — warning only.)

3. **U3 — Inline footnotes**.
   - Prompt teaches agent to emit `[^msg-<8-char-id>]` markers next to
     factual claims. Viewer renders natively.
   - Critique rule: `footnote_missing_on_metric` (warning) — if a claim
     contains a quantified metric (e.g., "99% reduction", "+7% CTR")
     and no footnote within ±1 line, warn.
   - Agent learns to own claim → source mapping as it writes.

4. **U4 — Revision style + archive mechanism**.
   - Prompt: when contradicting old info, rewrite the Summary and add
     a dated bullet to `## Recent changes`. For significant rollbacks,
     mint a `decisions/` page.
   - Viewer: render `<details>` HTML tags (mkdocs supports).
   - Critique rule: `strikethrough_detected` (blocker — user explicitly
     dislikes). If `~~` is in body, fail.

### V12-U5+ — medium scope (Wikipedia-style navigation)

Ships AFTER V12-tight lands and is validated. Orthogonal to prompt work.

5. **U5 — Backlinks ("Pages that link here")**.
   - Auto-generated at end of each page by coordinator post-batch.
   - Render as "## Linked from" section or footer chip list.

6. **U6 — Tag-based viewer filters**.
   - Frontmatter tags already exist. Viewer gets a "filter by tag" UI.
   - Cheap to add, huge reader UX improvement.

### V12-U7+ — large scope (tagging discipline, later)

7. **U7 — Proactive tagging**: prompt teaches agent to write 4+
   frontmatter tags per page (subject, domain, stakeholders, state).
   Reviewer enforces a minimum.

Defer until tight + medium show benefit. May not be needed if linking is
doing the work.

## Out of scope (explicitly deferred)

- **IM Reports group ingestion**: Amit's vision is eventually combining
  MP Launch (what / why / knowledge) + IM Reports (metrics / how-much /
  numbers) for the full company pulse. Deferred because IM Reports emails
  are image-heavy (tables, charts) and image parsing isn't in place.
  Revisit once MP Launch quality is satisfying.
- **Chatbot / query layer**: pages must be worth querying first. User
  question logs + feedback will become the best success signal but only
  after v12 lifts page quality. ~6 weeks out.
- **Image parsing**: separate ingestion pipeline work, not core v12.
- **Big-bang recompile of 300 existing pages**: compile-forward only.
  Trust self-healing — high-frequency concepts get fixed as their threads
  resurface; low-frequency ones don't matter.
- **Full 5348-email Cycle 10 scale**: at current $0.156/email, ~$278 to
  run. Needs cost reduction OR budget lift. Not a v12 goal.

## Philosophical takeaway (one line)

> If this system is working, the agent writes like a senior IndiaMart
> engineer explaining the business to a new joiner over coffee —
> grounded, linked, cited, with the constraints and history that make
> decisions make sense. If it's not, it writes like an email summarizer.
> v12 is the work to shift from the second to the first.
