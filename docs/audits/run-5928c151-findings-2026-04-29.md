## P0 findings

### F1 — Citation plumbing regression (50% of pages have refs without defs)

**What.** Half the touched pages (9 of 18) ship with `[^msg-xxx]`
inline references in the body but **zero or partial `## References`
section** holding the corresponding definitions. The reader has no
way to trace claims to source emails. mkdocs renders the body refs
as superscript links pointing to anchors that don't exist.

**Evidence.**
- `wiki/topics/im-insta-pro-auto-responder-matchmaking.md`: 23
  body references, 11 definitions → 12 dangling.
- 9 pages affected total (page-quality audit doc lists each).
- Validator did not catch this — its broken-wikilink check
  doesn't model footnote refs.
- Prior run-3e88f996 had 4.8% dangling, this run 50%.

**Root cause.** PR #260's commit message says *"the mkdocs hook
auto-builds `## References`, so writers no longer hand-author
it."* The prompt was updated to match that intent. But the mkdocs
hook (#258) only renders refs at *build time* — it does not write
the section back into the .md source. The agent stopped writing
the section, so the file on disk never contains the definitions
the reader needs to verify a claim.

**Proposed fix.** Coordinator-side deterministic backfill of
`## References`. Markers are stable Postgres data —
`message_id → raw path` is in `messages` table; every body
`[^msg-xxx]` slug maps to one and only one raw file. Add a hook
in `scripts/compile_all.py` post-batch (alongside the existing
`_normalize_touched_pages` step) that scans each touched
content-type page for body `[^msg-...]` references, finds the
matching message rows in Postgres, and rewrites the page's
`## References` section as the union of the prior section + any
newly-cited refs. Use a stable section-marker comment so the hook
is idempotent on re-run.

**Why coordinator, not prompt revert.** The bookkeeping is
deterministic, the agent reasoning budget is finite, and we just
spent it buying the lead-paragraph quality. Asking the agent to
also persist `## References` reintroduces the failure mode #260
was trying to retire. Coordinator wins on idempotence, cost, and
agent-attention budget.

**Severity.** P0 — citation plumbing is the core auditability
contract of this wiki. User flag: "super super important".

**Estimated cost.** ~80 LOC + tests. Single-file change in
`scripts/compile_all.py` plus one new helper.

## P1 findings

### F2 — Boy-scout clause (PR #260 line 324) is too narrow to repair adjacent rot

**What.** The boy-scout instruction was added in PR #260 at
`src/compile/prompts.py:324`:

> **Boy-scout / multi-page consolidation.** If you traversed
> multiple pages or links to find a fact, consolidate it: inline
> the answer or add a cross-link so the next agent doesn't redo
> the traversal. Leave the wiki better than you found it.

The clause as written gates on "If you traversed multiple pages
... to find a fact" and the prescribed action is "consolidate /
cross-link". It does **not** instruct the agent to:

- Fix existing broken wikilinks on pages it reads
- Refresh stale Summaries on pages it touches
- Dedupe parallel sections on pages it visits
- Backfill citations on previously uncited claims

**Evidence.** Page-quality audit found boy-scout score 0–1/3
across all 6 deep-read pages in this run, all greenfield.

**Why the clause underperforms.**

1. **Gated by traversal.** "If you traversed multiple pages" —
   most batches resolve their target page in one or two reads.
   The condition rarely fires.
2. **Action is consolidate, not repair.** The clause prescribes
   adding cross-links, not fixing broken ones. Even when it
   triggers, the agent reads it as "I should add a link", not
   "I should also fix the dead link I noticed at line 47".
3. **Reconnaissance-paralysis middleware (#251) actively
   discourages it.** The middleware nudges at 8 reads-without-
   write — exactly the pattern boy-scout exploration would
   produce. The two systems are working at cross-purposes.

**Proposed fix.** Three options, ranked:

A. **Extend clause + add explicit action verbs.** Replace
   line 324–327 with:
   > **Boy-scout / leave-it-better.** When you read a page you
   > didn't have to author today, scan its `## References` for
   > undefined `[^msg-...]` refs, scan body wikilinks for ones
   > that don't resolve, and scan `Recent changes` vs Summary
   > for staleness. Fix anything that takes <60s. The next agent
   > saves the same time twice.
   
B. **Coordinator-side staleness lint** (deterministic): on every
   touched page, the coordinator already has all the data it
   needs to detect broken wikilinks (validator does this) and
   undefined footnote refs (F1's hook). Surface these as a
   *warnings* file the next batch sees in its preamble. Doesn't
   require agent boy-scout behavior — pushes the work to a
   place that's already deterministic.

C. **Both.** A is cheap; B fixes the worst regressions
   regardless of whether the agent ever does A.

**Severity.** P1 — boy-scout was a stated PR-260 design goal
that did not land. User explicit ask: "How was this not fixed?
... Should have been?"

**Estimated cost.** Option A: 20 LOC prompt edit. Option B:
~60 LOC + tests. Option C: both, additive.

### F3 — `qmd_timeout_s` 45s → 60s

**What.** `src/config.py:182` sets `qmd_timeout_s: int = 45`.
The trace audit observed 15.4% of `resolve_page.semantic` calls
hit the 45s timeout (vs 8.9% in run-3e88f996, +6.5pp).

**Evidence.**
- `src/config.py:179-182`:
  > `# qmd_timeout_s caps the per-call subprocess wall-clock`
  > `# (45s covers ...)`
  > `qmd_timeout_s: int = 45`
- Trace audit reports 15.4% timeout rate.

**Proposed fix.** Bump to `60`. Embedding service has
degraded since 3e88f996; current run shows 30s+ p90 calls
landing inside budget but 45s+ calls being killed.

**Severity.** P1 — direct cost driver (timeouts trigger
recursion-limit-adjacent retry loops).

**Estimated cost.** 1 LOC + test fixture update.

### F4 — Mid-run pool quarantine (#252) silent failure on 429 burst

**What.** `glm-5.1` returned `429 No deployments available` on
batch 4, killing all 5 emails. The mid-run pool quarantine
(#252) did not exclude `glm-5.1` from subsequent batches —
glm-5.1 served batches 8, 9, 10, 13, 14, 16, 17 after the
burst.

**Evidence.** Live log + trace audit. `_healthy_pool` 4h
rolling threshold needs ≥50% of attempts in window to fail
to quarantine (currently 5/76 = 6.5%, well under). One
glm-5.1 burst doesn't move the rolling rate enough.

**Proposed fix.** Add a **consecutive-429 short-circuit** to
`_healthy_pool`: if any model has ≥1 `429 No deployments`
in the last N=3 attempts, soft-quarantine it for 10 minutes.
This is a complement to (not replacement for) the rolling
threshold.

**Severity.** P1 — silent failure mode. The middleware ran
and returned "healthy", which is misleading.

**Estimated cost.** ~20 LOC + test.

### F5 — StuckHeartbeat (#253) wrong layer for OTel-export inter-batch stall

**What.** Inter-batch quiet windows (>2 min) are happening with
`Failed to export span batch` (Langfuse OTel exporter)
preceding the silence. StuckHeartbeat (#253) operates intra-
round (per-LLM-call) and never fires for these stalls.

**Evidence.** Trace audit found `Failed to export span batch`
preceding the inter-batch quiet that triggered the false
"wedged" alarm. Process is alive; only the dispatch loop is
slow.

**Proposed fix.** Add a **dispatcher-level wall-clock watchdog**
in `scripts/compile_all.py` outside the per-batch invoke. If
no batch starts within 5 min of the prior batch's `finished_at`,
surface a structured warning + dump the running thread stacks.

**Severity.** P1 — diagnostic, not fatal. But makes
"is the run wedged" a 0.5s eyeball check instead of guesswork.

**Estimated cost.** ~30 LOC + test.

### F6 — Heuristic scorer ⊥ LLM judge (Pearson r = −0.758)

**What.** The heuristic scorer (`scripts/score_wiki.py`) and the
LLM judge (`scripts/judge_wiki.py`) **anti-correlate** on the
n=5 sample from this run. The heuristic top-2 (8.6, 8.4) were
the judge's mid-tier (7.00 each); the heuristic bottom (6.4)
was the judge's top (7.67).

**Evidence.** `docs/audits/run-5928c151-llm-judge-2026-04-29.md`
section "Heuristic vs Judge correlation".

**Why it matters.** The heuristic is the cheap signal we use
to triage which pages need a deeper look. If it inverts judge
ranking, it's actively misleading — we'll spend audit budget
on pages judges think are fine and ignore the ones they hate.

**Diagnosis.** `concept_shape` and `source_density` columns
saturate near 10 (no discrimination). `structural_smells`
column appears to penalize pages judges actually like
(verbose Summaries, long sections — but with content).

**Proposed fix.** Rebalance the scorer:
- Drop or floor-normalize the saturated columns
- Make `structural_smells` a *report* column, not a *score*
  column
- Add a "References completeness" column (zero-cost — just
  `len(re.findall(r'\[^msg-)') / len(definitions)`). This
  alone would catch F1.

**Severity.** P1 — affects every future audit decision. Cheap
to fix.

**Estimated cost.** ~50 LOC + new test fixtures.

### F7 — Engineer ramp-up: structured `engineering_surface:` frontmatter

**What.** Persona audit found 8/8 pages still lack repo paths,
GCP projects, namespaces, alert names, or kibana shortlinks.
PM gains came from extracting one structured field (`owner:`).
Engineer gains require the same pattern.

**Evidence.** `docs/audits/run-5928c151-personas-2026-04-29.md`.

**Proposed fix.** Add an optional structured frontmatter
block:
```yaml
engineering_surface:
  repos: ["repo-name1", "repo-name2"]
  services: ["service-name"]
  alerts: ["alert-rule-id"]
  dashboards: ["grafana-uid"]
```
+ a critique rule: for any topic page tagged `domain:
engineering` (or otherwise system-adjacent), require at
least 2 entries across `repos`/`services`/`alerts`. Same
pattern as the `owner:` extraction that won the PM round.

**Severity.** P1 — missing piece for engineer ramp-ready
score >= 2.

**Estimated cost.** ~40 LOC prompt + 30 LOC critique +
fixtures.

## P2 findings

### F8 — Grok-4.1-fast under-picked (4/21 = 19% live, was 0/17 in trace audit)

**What.** Random pool selection is uniform `random.choice(pool)`
across 3 entries. Expected pick rate = 33%. Observed at
batch 21 = 19%. Trace audit caught 0/17 in the first window.

**Diagnosis.** No code bug — `random.choice(pool)` is uniform.
On n=21 with p=1/3, observed 4 picks gives p-value ≈ 0.10
(binomial tail), so this is *plausible* random variance but
suggestive. Would be definitive at n>50.

**Proposed action.** Wait for run completion (n=75) and
re-measure. If still ≤25%, investigate `_refresh_pool_for_
batch` for any silent-exclusion bug. If converges to 33%,
close as small-sample noise.

**Severity.** P2 — wait-and-see.

**Estimated cost.** 0 (observation only) until escalated.

## Headline

The prompt edits in #259 + #260 decisively moved the
PM-forwardability signal but introduced a citation-
plumbing regression (F1) that affects half the corpus and
inverts the wiki's verifiability contract. Boy-scout
behavior was added to the prompt (line 324) but with

high-value. Heuristic scorer (F6) is misleading until
rebalanced. Engineer ramp-up needs structured frontmatter
(F7) — same pattern as the PM win.
