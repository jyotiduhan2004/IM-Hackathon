## RCA + sizing + ideas (per finding)

For each finding below: **RCA** = why it happens, **Size** = how much it costs / how often, **Ideas** = ranked fixes from cheapest/safest to deeper.

### B1 — `(preserve existing)` placeholder leak  [P0]

- **RCA.** The agent hallucinated `... (preserve existing)` as an edit-time placeholder. The string itself isn't in `src/compile/prompts.py` — but the prompt uses the word **"existing"** 8+ times as instruction text (L91, L416, L554, L557, L657, L659, L716). The LLM has adopted "existing" into its working vocabulary and emits placeholder phrasing like `... (preserve existing)` when it wants to indicate "don't rewrite this". Classic instruction-as-content failure: the agent transcribes its mental note instead of executing it.
- **Size.** 2 / 42 pages = 4.8% of run. Visible content corruption in deployed wiki. The two pages were both `edit_file`-style updates to existing content (gladmin SOA section + agentic-auditor References block). Risk model: any agent doing partial-section edits is exposed.
- **Ideas (ranked).**
  1. **Cheapest:** add a `validate_wiki` blocker for the literal substrings `... (preserve existing)`, `<TODO>`, `[insert ...]`, `[copy existing]`, `[keep ...]`, `…(unchanged)` (case-insensitive). 1 regex + a test. Catches the leak at the gate.
  2. **Tighter (recommended):** include the same regex in `find_touched_pages` (critique-time) so the agent must self-correct in the same batch instead of shipping the leak. Treats it like the duplicate-H2 / undefined-footnote checks already do.
  3. **Deeper:** rewrite the prompt's "edit an existing page" sections to use directive verbs ("update Section X by adding…") instead of preservation language ("keep existing"). Removes the linguistic source — but riskier; the prompt rewrite itself can regress other behaviors.

### B2 — Empty frontmatter `{}` shipped on 2 pages  [P0]

- **RCA.** Task #180 was supposed to fix this — "google-ads page empty frontmatter" landed weeks ago. But two new pages from this run still shipped with `--- {} ---`. Hypothesis: #180 patched the *catalog-sync* path but not the *initial write* path. The agent's `write_file` can land a page with empty frontmatter, the validator flags it, but the file is on disk before validation runs (validator is a critique, not a write-gate). The catalog-sync now refuses these (#180), but the page ITSELF is committed to wiki/.
- **Size.** 2 / 42 pages = 4.8%. These pages are invisible to nav rollups (`page_type` is required to render in domain pages and home), so they effectively don't exist for navigation. Higher impact than count suggests.
- **Ideas.**
  1. **Cheapest:** validator-blocker mode in `validate_wiki` (block publish on `{}` frontmatter). Already partly there — verify it actually blocks, not just warns.
  2. **Right fix:** the `write_file` / `edit_file` tool wrappers should refuse to write a page whose YAML frontmatter is `{}` or missing required fields (slug, page_type, status). Reject before disk hit, with a concrete error pointing the agent at the required schema. Mirrors `EditPayloadSanityMiddleware`.
  3. **Investigation:** find the test that #180 added and check whether it covers the `write_file` path or only the catalog-sync path. If only the latter, expand it.

### B3 — `glob` 68% timeout (24 min/run waste)  [P1]

- **RCA.** The agent calls `glob("**/<slug>.md")` to find a wiki page by slug — exactly the contract `resolve_page` and `get_page_summary` were designed to replace. `GlobNarrowingMiddleware` exists (PR #126) and rejects `**/<slug>.md` patterns, but the trace audit shows 71 calls slipped through — meaning either the middleware list ordering puts glob before the narrower (and the narrower never sees them), or the patterns the agent uses are slightly different shapes that the middleware regex doesn't match. The trace audit shows shapes like `**/auditmate.md` and `wiki/**/auditmate.md` — both should be caught.
- **Size.** 71 calls × 20 s = **23.6 min of pure timeout latency per run** (~10% of total runtime). Multiplied across all daily compile runs, this is the biggest single wall-clock-cost recovery opportunity in the codebase.
- **Ideas.**
  1. **Cheapest:** read `src/compile/middleware/glob_narrowing.py`, identify which slug-shape patterns slip through, broaden the regex. Should be a 5-line PR.
  2. **Stronger:** make `glob` reject ANY pattern containing `<slug>.md` at the tail (any prefix), not just specific shapes. Override the deepagents default with our own narrower tool that hard-fails with a pointer to `resolve_page`.
  3. **Deepest:** retire `glob` entirely from the agent toolset. Audit (PR #118) was inconclusive on whether the reviewer subagent still needs it — verify, then delete.

### B4 — People-slug wikilinks in 76% of pages  [P1]

- **RCA.** The agent generates wikilinks freely as it writes. `CLAUDE.md` says people pages are reference-only and shouldn't appear in body/Related, but the agent has no automatic guard. The prompt has been told "wikilink people only when a topic genuinely needs them" — agent drifts. Worst page (`msite-pdp-html-rewrite-phase-2-...`) has 71 leaked links, suggesting the agent dumped the entire stakeholder list into Related.
- **Size.** 32 / 42 pages = 76%. Single biggest content-quality issue. The page-quality scorer caps the `structural_smells` penalty at -4, so the worst offender (71 leaks) scores the same as a page with 5 leaks — the heuristic is flat-lining. Director-readiness collapses because pages look like email-thread participant dumps, not synthesized concepts.
- **Ideas.**
  1. **Cheapest (recommended):** coordinator post-processor — when an agent writes `[[<email-canonical-slug>]]` that resolves to `wiki/people/<slug>`, replace it with the display name (from the people page frontmatter) in body sections. Don't ask the agent to remember; deterministically clean it up before the page hits disk. ~50 LOC.
  2. **Stricter:** treat the leak as a critique-time blocker. Agent must self-correct or the batch fails. Riskier — could cause batches to fail when the substantive content is fine.
  3. **Schema:** add `participants:` as a structured list field (separate from `Related`). Renderer shows them at the bottom of the page; agent can't put them in prose.

### B5 — Glossary actively misleading  [P1]

- **RCA.** `wiki/glossary.md` is auto-generated by regex against all email/wiki content. Regex picks up 2-3-letter sequences without context. No human-curated baseline; everything is auto-derived. Once a wrong definition lands ("AI = GenUI Agent"), it sticks.
- **Size.** Single file, ~50+ entries, hard to estimate "% wrong" without manual review — but the engineer-persona auditor flagged 3 obviously-wrong entries in the first pass, suggesting a high error rate. Whole file is suspect.
- **Ideas.**
  1. **Cheapest:** delete the auto-generation. Hand-curate 30-40 IndiaMart acronyms in a single PR (MCAT, ISQ, BL, BuyLeads, NPS, OBD, FCP, HPA, PNS, SOI, etc.). Add CI gate that fails if the file is regenerated automatically. Freeze.
  2. **Hybrid:** keep auto-generation but require manual approval for every new entry. Glossary entries become reviewable PRs.
  3. **Defer:** do nothing. The glossary is one file and few users land there. The cost/value of fixing is low compared to B3 / B4.

### B6 — No owner / DRI in frontmatter (0/9 pages PM-forwardable)  [P1]

- **RCA.** Frontmatter schema doesn't have ownership fields. Agent extracts ownership info inconsistently into prose with section names like `Stakeholders`, `Key Contacts`, `Development Team`, `Leadership Feedback`, `Owner` — no convention. PMs can't grep for ownership.
- **Size.** 0 / 9 pages have it. 100% of compiled pages are PM-unforwardable to a director without rewrite. Big structural gap; affects the wiki's "trust-and-share" property.
- **Ideas.**
  1. **Cheapest:** add `owner:`, `dri:`, `pm:`, `eng_lead:` (all optional) to the frontmatter schema. Update validator to allow them. Update one example in the prompt that shows their use. Don't make them required yet — observe whether the agent populates them naturally over a smoke or two.
  2. **Stricter:** add a critique-time check that flags pages with stakeholder-prose-but-no-frontmatter-fields. Nudge to extract.
  3. **Renderer fix:** mkdocs override that pulls `owner:` / `dri:` to a colored badge above the lead paragraph. Visible PM-friendly affordance.

### B7 — Duplicate Related in 79% of pages  [P2]

- **RCA.** Frontmatter has a `related:` list AND the body has a `## Related` H2 list. They drift apart because the agent edits one but not the other. Originally a renderer convenience (frontmatter populated mkdocs nav) but body Related was kept for visibility.
- **Size.** 33 / 42 pages = 79%. Drift is cosmetic but signals "no source-of-truth".
- **Ideas.**
  1. **Cheapest:** pick one source-of-truth (frontmatter), drop the body section. mkdocs override that auto-renders the body Related list FROM frontmatter at view time. ~30 LOC + a one-time migration.
  2. **Defer:** acceptable if neither side blocks deploy.

### B8 — Pages with 0 footnotes despite many sources  [P2]

- **RCA.** V12-U3 (PR #142) added inline `[^msg-abc]` footnotes. The agent doesn't always add them. No critique-time floor.
- **Size.** 2 / 42 pages = ~5% of run. Affects audit/traceability of those specific pages.
- **Ideas.**
  1. **Cheapest:** existing pending task #162 — citation density floor in critique. Block batches where footnote-per-50-lines < 0.5 on any touched page. Already on the backlog.

### B9 — `check_my_work` loops on glm-5  [P2]

- **RCA.** glm-5 calls `check_my_work` median 4×/batch (max 16× in batch 41 for a 1-page edit). Grok median is 1. Either prompt encourages "verify after every patch" or the model has internal bias to over-verify. PR #168 added per-write-epoch caching (idempotent cmw), so repeated calls without a write between SHOULD return cached. Either the cache isn't hitting (epoch bumping wrong) or the agent is interleaving cmw with edits aggressively.
- **Size.** Each cmw call is a full LLM round (~5-10s). 16× ≈ 80-160s amplifier on a single batch. ~5-10% cost increase on glm-5 batches. Doesn't fail anything; just expensive.
- **Ideas.**
  1. **Cheapest:** verify the idempotent cache is firing — log the cache hit/miss in `_check_my_work_cache` lookups and re-run a small smoke. If hits are happening but glm-5 is still calling 16×, then we're paying for the call regardless of cache (cache eats the LLM round, not the call cost? — verify).
  2. **Stricter:** hard cap in `CheckMyWorkGateMiddleware` — third call without a write between returns the cached value AND emits a strong "stop verifying" tool message.

### B10 — No observation has `level=ERROR` despite 129 tool errors  [P2]

- **RCA.** Tool wrappers return `{"status": "success", "content": "Error: ..."}` — they don't elevate to error. LangChain ToolNode never marks the observation `level=ERROR` because `status="success"`. So Langfuse error-rate dashboards under-report by definition.
- **Size.** Observability quality issue. 129 silent tool errors per run = 0% surfaced. Doesn't break anything, but breaks debugging.
- **Ideas.**
  1. **Cheapest:** tool wrappers (especially `glob`, `edit_file`, `read_file`) should set `status="error"` when their output starts with `Error:`. ~3-line change per tool.
  2. **Wider:** add a Langfuse callback that lifts errors automatically when output starts with the prefix. Catches it for any tool past, present, future.

### B11 — Trace `tags` empty at top level (filter broken)  [P3]

- **RCA.** `compile_all.py` sets `["email-kb", "compile", "model:<m>", "batch:<n>"]` in `config["tags"]`. The Langfuse SDK or `get_langfuse_handler` puts them in `metadata.tags` instead of top-level `tags`. Either langfuse-langchain v3+ changed the API or the handler config is wrong.
- **Size.** Filter-by-tag in Langfuse UI/API for this run returns zero. Debugging inconvenience, not a correctness bug.
- **Ideas.**
  1. **Cheapest:** read the langfuse-langchain v3+ docs on tag propagation, fix the one-line config. Most likely needs `tags=` passed to the trace creation, not the trace config.

### B12 — Wiki is engineer-grade, not PM-grade (0/9 pages forwardable)  [structural]

- **RCA.** Compile prompt and reviewer subagent both have engineer-leaning bias: technical accuracy, citation completeness, structural lint. Neither has a "would a director get value from this in 30 seconds?" check. The 5W heuristic exists in the prompt (V12-U2 / PR #141) but is content-vague — agents satisfy it with engineer-domain answers (which API, which deployment) rather than PM-domain answers (what's the open ask, by when, scaling to whom).
- **Size.** Affects the wiki's stakeholder usefulness. Hard to quantify; a director-share rate of 0% is the headline.
- **Ideas.**
  1. **Defer to v13.** This is a structural reframing, not a PR. Worth a separate proposal.
