# qmd Phase 0 spike — GO

**Decision: GO.** qmd's hybrid BM25 + vector + LLM-rerank pipeline returned sensible top-5 candidates on 34/40 (85%) real Langfuse resolve_page queries, beating the 80% threshold set in `docs/proposal/QMD-INTEGRATION-PLAN.md`. Fixture backstop: 9/9 questions produced sensible top-5 pages. Proceeding to Phase 1.

## Headline numbers

| Metric | Target | Observed | Pass? |
|---|---|---|---|
| Langfuse sample top-5 sensibility | ≥80% (≥24/30) | **85%** (34/40) | ✅ |
| Fixture top-5 sensibility | ≥7/9 | **9/9** | ✅ |
| Fixture exact-slug hits (where page exists) | (informational) | 3/3 exact, 6/6 related-topic surfaced | — |
| qmd p50 latency | (no gate) | **12.34s** | — (warm, rerank-on) |
| qmd p95 latency | (no gate) | **20.88s** | — |
| SQL p50 latency | (baseline) | 37ms | — |
| SQL p95 latency | (baseline) | 53ms | — |

qmd is ~330× slower per-call than SQL-ILIKE. Expected and acceptable per the decided plan — the quality lift is the point, and Phase 2 will measure whether fewer retries per concept offset the higher per-call cost.

## Setup

- **qmd version**: 2.1.0 (installed via `bun install -g @tobilu/qmd`)
- **Models used**: embedding-gemma-300M-GGUF, Qwen3-Reranker-0.6B-Q8_0-GGUF, qmd-query-expansion-1.7B-gguf (all cached in `~/.cache/qmd/models/`, ~2GB total)
- **Hardware**: Apple M4 Pro, Metal GPU offload enabled
- **Corpus indexed**: 416 non-person wiki pages across 5 collections (`wiki-topics` 305, `wiki-systems` 100, `wiki-decisions` 1, `wiki-policies` 1, `wiki-domains` 9). People pages (`wiki/people/`, 104 files) explicitly excluded per the decided plan.
- **Embed wall time**: 47s for 1,056 chunks from 416 documents (first-run model download not included).
- **Search mode**: `qmd query` (hybrid, default reranking on, default query expansion on).

## Method

1. Extracted 440 real `resolve_page` tool invocations from 100 recent Langfuse traces; deduped to 325 unique queries; randomly sampled 40 (seed=42). Script: `scripts/spike_extract_resolve_queries.py` (one-off, to be deleted before merge).
2. For each sampled query, ran `qmd query <q> -n 5 --json` and `src.compile.tools.raw_access.resolve_page(q, limit=5)`. Comparison runner: `scripts/spike_compare.py` (one-off, also delete).
3. Eyeballed every query's qmd top-5 by hand. Label = "sensible" if at least one of the 5 is plausibly on-topic; "not_sensible" otherwise. No exact-match requirement; the agent's LLM judgement disambiguates after we surface candidates.
4. Ran 9 fixture questions from `tests/fixtures/new_joiner/domains.json` as an independent backstop.

## The 6 "not sensible" cases

### Failure mode 1 — Person-email queries (4 cases: #3, #9, #22, #28)

Queries: `monarch-hasija-indiamart-com`, `sayan-samanta-indiamart-com`, `shreyans-singh-indiamart-com`, `aditya-rai-indiamart-com`.

qmd returned unrelated topic pages because we explicitly excluded `wiki/people/` from the index. SQL returned the correct person page. This is **by design, not a qmd failure** — the canonical path for people is `create_entities(...)`, not `resolve_page`. But the trace data shows the agent is calling `resolve_page` for people today — that's a tool-usage issue.

**Fix path (Phase 1 or 2)**: docstring guidance ("for people, use `create_entities`"), OR the user-suggested "find_exact vs find_similar" tool split, OR a silent re-route in the shim when the query matches an email shape (`*-indiamart-com`) → call `create_entities` path.

### Failure mode 2 — Exact system-slug miss (#11: `marketplace-launch`)

Query `marketplace-launch`. The exact page `wiki/systems/marketplace-launch.md` exists. qmd top-5: `foreign-bl-purchase-mail-ownership-transfer`, `replacing-and-removing-expired-bl-alert-notifications`, `marketplace-launch-csl-clickstream-migration-aws-to-gcp`, `soa-indiamart`, `enhanced-hrs-user-identification-aadhar-udyam`. The direct system page is missing from top-5 entirely; top-3 is a child topic (`-csl-clickstream-migration-`) that shares the prefix.

Hypothesis: rerank over-weighted semantic prose about "marketplace launch" work items vs the short-slug system page. Possibly a corpus-specific issue — the `marketplace-launch.md` system page's body is mostly a sources list (see the earlier read), so rerank saw thin text content.

**Fix path**: not blocking for Phase 1. Worth watching in Phase 2 traces. If this pattern recurs (short-slug queries missing their direct system page), consider a "exact-slug fast path" pre-check before qmd — a 2-line lookup that short-circuits if the query literally matches a known slug.

### Failure mode 3 — Long slug query miss (#23: `msite-pdp-html-rewrite-phase-2-dom-simplification-size-reduction`)

The query IS the exact slug of a page in `wiki/topics/`. qmd returned related page-rewrite topics (`marketplace-discovery`, `msite-company-home-page-html-rewrite`, `desktop-pdp-css-rewrite-llm`) but not the exact page. SQL correctly returned the exact match.

Hypothesis: query expansion broke the slug into semantic variants, and the resulting multi-query hybrid diluted the exact match.

**Fix path**: same as #11 — a short-circuit "exact slug match" check before handing to qmd. This is a small, additive code path we can bolt onto the qmd_client shim in Phase 1 if the issue recurs. Don't over-engineer upfront; measure first.

## Latency distribution

Per-call latency (qmd, hybrid + rerank, warm GPU):

- p50: 12.34s
- p95: 20.88s
- max: 21.98s
- min: ~5.3s (simpler queries, faster rerank)

SQL-ILIKE baseline for contrast: p50 37ms, p95 53ms.

Per-query breakdown (observed from `qmd query` progress lines during the spike):
- Query expansion (Qwen3 1.7B GGUF): ~1.2s
- Embedding the query (embedding-gemma-300M): ~300ms
- BM25 + vector retrieval + RRF: fast (<500ms)
- Reranking 40 chunks (Qwen3-Reranker-0.6B): **~8-16s** (dominant cost)

**Implication for Phase 1**: if latency becomes a problem in production, `--no-rerank` drops p50 to ~2-3s at some quality cost. We're starting with rerank ON per the decided plan; revisit if Phase 2 traces show the latency bothers compile throughput.

## Observations worth carrying into Phase 1

1. **Rerank is the latency hotspot** (~8-16s per call). If we hit latency ceiling, `--no-rerank` is the first knob.
2. **Short-slug exact matches sometimes miss top-5** (#11, #23). A pre-qmd exact-slug short-circuit would patch this cheaply. Consider after Phase 1 traces confirm it's a pattern.
3. **Person-email queries are real in traces (~10%)**. The agent is using the wrong tool. Docstring guidance or a sil-re-route hook in the qmd shim is the cleanest answer. The "find_exact vs find_similar" two-tool split idea also addresses this naturally.
4. **Query-shape sensitivity**: qmd's sensibility on prose / multi-word / ALL-CAPS queries is 100% in the sample (16/16). The only qmd-specific failures were on exact-slug shapes that still exist (#11, #23). Ironically, qmd is BETTER at the "hard" queries than the "easy" ones — the opposite of SQL-ILIKE's failure mode. This is a good complementary profile; a tiny exact-match pre-check captures the residual.
5. **SQL returned empty for `leadmanager` (#17)** — qmd returned `lead-manager` at #2. qmd handled a typo-ish query SQL couldn't. Representative of the win.

## Recommendation

**Proceed to Phase 1** per `docs/proposal/QMD-INTEGRATION-PLAN.md`. Priority adjustments from spike findings:
- **Add a pre-qmd exact-slug short-circuit** to the qmd_client shim. 10 lines, catches failure modes #11 and #23. If the query is a kebab-case-only string and matches an existing slug exactly, skip qmd and return the direct lookup. Matches the "two-tool split" idea in spirit without the larger API change.
- **Keep person-email handling as Phase 2 docstring work**. Don't bolt it onto Phase 1 — it's a cross-tool agent-guidance issue, not a retrieval issue. Revisit once we have docstring-revision traces (per the measurement-first plan).
- **Instrument rerank-latency specifically** in the Phase 1 Langfuse span (not just total latency) so we can triage latency issues without re-spiking.

Everything else in the decided Phase 1 plan stands.

## Follow-up findings (same session, after GO)

After the initial GO call, ran targeted tests to nail down two outstanding questions. Results below; the full set of ~77 queries from this session is saved at `docs/audits/qmd-spike-2026-04-23-queries.jsonl` as a reusable regression corpus.

### qmd indexes full page body content (verified)

Body-only distinctive phrases query correctly to their source page:

- **"Wrong Mapping disposition"** → `contextual-blni-feedback` top-1 score 0.93; snippet at line 37 showing the exact body text.
- **"seller-MCAT relevance A and BA rank"** → `contextual-blni-feedback` top-1 score 0.93; snippet at line 48.
- **"Ticket 636565"** → `export-get-quote-form-enrichment` top-1 score 0.93; snippet shows line 64 where the ticket ID appears in structured metadata.

qmd's snippets include line-numbered excerpts (`@@ -48,4 @@` format, ±N lines configurable). This is free output — we should pass snippets through in `resolve_page`'s response so the agent sees WHY a page matched without a full file read. **Add to Phase 1: include `snippet` field per candidate in the resolve_page return envelope.**

### BL → buylead semantic mapping works (learned from corpus, no hardcoded list)

"BL recommendation engine" query → top-5 all BL/buylead-related pages. Our corpus interleaves both spellings ("WA9696 BuyLeads (BLs)"), so the embedder learned the mapping implicitly. Same applies to ISQ / PMCAT / BLNI / MCAT acronyms.

### Deterministic lookups — strong on prefixed phrases, weak on bare tokens

15-query deterministic test showed a clear pattern:

**qmd strong on**:
- Ticket IDs *with prefix* ("Ticket 636565" → exact top-1)
- Acronyms defined in corpus ("PMCAT", "ISQ" → exact top-1)
- Version strings with context ("Android v1369" → exact top-1)
- Full-qualified domains / URL fragments

**qmd weak on**:
- Bare numbers ("638983" alone) — rerank surfaces unrelated pages at high scores
- Code-like identifiers ("eto_lead_pur_hist" DB table) — embedder appears to fail tokenization; returned thematically-adjacent pages, not the actual match
- Isolated timestamps ("15:37 IST")
- Very short single-token queries ("BLNI" alone returns an index page at top-1)

**Implication**: we keep `grep` as a separate tool for "find any page that literally contains string X". qmd is the concept-search tool; grep is the string-search tool. Phase 1 note: don't deprecate or re-route grep; the two are complementary.

### Revised Phase 1 design hooks (additive to the original plan)

1. **Response envelope: add `snippet` field per candidate** (3-line change). qmd computes them already; pass through.
2. **Exact-slug short-circuit, generalised**: the original idea (if query is a kebab-case token matching an existing slug, skip qmd and return direct lookup) naturally extends to cover the "bare number" and "code identifier" weak cases — same short-circuit, different shape check.
3. **Instrument rerank-latency separately** in the Langfuse span — rerank is 8-16s of a 12s p50 call. Triageable as its own metric.
4. **`tests/fixtures/qmd_spike/` regression corpus**: seed from this doc's queries JSONL when Phase 1 adds integration tests. 77 queries across 5 shape categories — enough to detect regressions on ranking-model changes, query-expansion toggles, or reranker bumps.
