## 2. The single biggest UNSPOKEN risk in current trajectory

**Dual-write drift between raw files and Postgres.**

The current repo still ingests by writing markdown only:
`scripts/ingest_backlog.py:139-163`, `scripts/watch_and_compile.py:97-109`,
and `src/ingest/parser.py:136-153` produce raw files with `compiled: false`
in frontmatter. The current queue still reads and mutates that frontmatter
in `src/compile/compiler.py:24-59` and `132-169`. Operational tooling
still trusts the raw flags too: `scripts/compile_overnight.sh:55-61`,
`scripts/stats.py:54-65`, `scripts/dipstick.py:231-242`, and
`scripts/snapshot_wiki.py:116-137`.

If the slice ships exactly as described and ingest is not changed
immediately, the first new email after backfill creates a raw file
humans can see and a queue row Postgres may not know about. Then every
report disagrees. That failure will look like "Postgres lost mail," even
when the schema is fine.

> Status as of merge: the messages PR added a best-effort
> `_register_in_catalog()` hook inside `parser.write_raw_email()`, so
> both `ingest_backlog.py` and `watch_and_compile.py` get the DB write
> for free. PR1 in the roadmap above is therefore down to "audit + tests
> for that hook", not net-new code.

## 3. Code hotspots that will become much harder once messages live in Postgres

- `src/compile/compiler.py:24-59`, `132-169`, `407-440`. This file is
  the public queue contract. `list_uncompiled_emails()` and
  `mark_as_compiled()` still expose raw-file semantics to the agent,
  and `run_compilation()` has no transaction boundary around "wiki
  changed" versus "message finished."
- `src/compile/prompts.py:8-10`, `41-45`, `54-58`, `185-215`. The model
  is explicitly told that raw files are mutable queue state and that
  `sources:` must be rebuilt by grepping raw mail. After the slice,
  that prompt is wrong and will actively fight the migration.
- `scripts/compile_all.py:100-209`, `scripts/compile_parallel.py:88-136`,
  `scripts/watch_and_compile.py:115-145`. Three separate schedulers
  importing the same raw queue tool. Patching only one means three
  different queue behaviors.
- `scripts/backfill_stubs.py:149-243`. `--recompile` currently flips
  raw `compiled` back to `false` to force work. After Postgres becomes
  truth, that becomes hidden divergence unless it enqueues `message_id`
  or `page_id` in the DB instead.
- `scripts/compile_overnight.sh:55-61`, `scripts/stats.py:54-65`,
  `scripts/dipstick.py:231-242`, `scripts/snapshot_wiki.py:116-137`.
  Every one still counts or resets work by reading `compiled: true`
  from raw files. `scripts/snapshot_wiki.py:123-133` also still uses
  `split("---", 2)` even though `src/utils.py:1-39` documents why that
  parser is broken.
- `src/ingest/parser.py:136-153` plus callers in
  `scripts/ingest_backlog.py` and `scripts/watch_and_compile.py`. Where
  the second source of truth is born. Leaving queue semantics in
  serialized frontmatter here for long makes every later migration
  harder.

## 4. Honest week-1 deliverable assessment

Realistic week-1 outcome at the current pace: an owner-operated internal
pilot with a safer compile queue, a static MkDocs site, and a subset of
strong topic pages for recent work. Matches the repo's own phase split:
Phase 1 is live ingest, Phase 2 is search and better UI
(`docs/issues/07-phased-delivery.md`). Current browse surface is still
only MkDocs Material plus the built-in `search` plugin
(`mkdocs.yml:72-75`); no query API at all (`src/api/__init__.py` is
0 bytes).

**No, this is not on track yet for someone other than the owner to
browse/search successfully without guidance.** Evidence already in the
audits:

  stubs; 96 pages are effective orphans

  describes the wiki as "a promising pile"

Ship week 1 as "owner can recover recent decisions faster from compiled
topic pages." Do not sell week 1 as "team-ready browse and search."
