## 1. Compiler-registered tools (actually passed to `create_deep_agent`)

Source: `src/compile/compiler.py:374-382`. Six tools are registered. All are pure file-backed: no DB, no network.

### 1.1 `list_uncompiled_emails(raw_dir: str = "raw") -> list[dict[str, str]]`
Defined `compiler.py:23-59`. Returns `[{path, date, subject, from, thread_id}]` per uncompiled raw email, sorted oldest-first.

| Anthropic criterion | Grade | Notes |
|---|---|---|
| Structured, useful response | B+ | Keyed dict with natural-language fields (`date`, `subject`, `from`), no cryptic IDs — matches the post's "contextual relevance over flexibility" rule. |
| Surfaces errors with actions | D | Silently `return []` when `raw/` missing. A new user would think there is nothing to do. Should return an error dict the agent can react to. |
| Validates before side-effects | N/A | Read-only. |
| Echoes state for next decision | C | Does not include total raw count, compiled count, or thread grouping. The agent immediately re-derives thread grouping itself (see `compile_all.py:38-70`), wasting tokens. |
| Right level of abstraction | A- | Matches the Anthropic "fewer, high-impact tools" principle — one call replaces `ls raw/` + `read_file` × N. |
| Composes with context limits | B | Unbounded — 300+ raws would return 300 dicts. No `limit` / `offset` / `since` filter. |

Improvements:
- Add `limit`, `offset`, `since` (ISO date), `thread_id` params. Default `limit=50`.
- Return `{"total": N, "returned": M, "truncated": bool, "emails": [...], "threads_represented": K}` so the agent has pagination signal up front.
- On missing `raw/` return `{"ok": false, "error": "raw_dir not found: raw", "hint": "Run scripts/ingest_backlog.py first"}`.

### 1.2 `list_wiki_pages(wiki_dir: str = "wiki") -> dict[str, list[str]]`
`compiler.py:62-90`. Returns `{topics: [...], entities: [...], systems: [...], policies: [...], timelines: [...], conflicts: [...]}` — filename stems only, no metadata.

| Criterion | Grade | Notes |
|---|---|---|
| Structured/useful | C | Flat filename list. Agent cannot tell `arjun-gaur` from `arjun-gaur-clean` from `arjun-gaur2` by purpose — exact pain #1 in the synthesis ("IA audit counted 11 near-duplicate pairs"). |
| Error surfacing | F | Silently returns empty categories. |
| Validates before side-effects | N/A | Read. |
| Echoes state for next decision | D | No titles, no status, no size, no last_compiled, no backlinks. The agent still must `read_file` every candidate to disambiguate — the exact "brute-force" address-book anti-pattern Anthropic calls out. |
| Right abstraction | C | Reads like `list_contacts`. Should be `search_wiki_pages(query)` / `find_similar_pages(slug)`. |
| Composes with context | D | 461 stems × ~20 chars ≈ 9 KB eaten on every call. The compiler is **told by the system prompt to call it "once at the start"** — so this gets paid every batch. |

Improvements — and this is the highest-leverage single change in the audit:
```python
@tool
def list_wiki_pages(
    wiki_dir: str = "wiki",
    response_format: Literal["concise", "detailed"] = "concise",
    category: str | None = None,
) -> dict:
    """Concise mode: counts per category + first 20 per bucket.
    Detailed mode: {slug, title, status, size_bytes, last_compiled} per page.
    """
```
Example concise return (~600 tokens):
```json
{
  "counts": {"topics": 96, "entities": 307, "systems": 58, "policies": 0, "timelines": 0, "conflicts": 0},
  "recently_updated": [{"slug": "dynamic-smart-rfq-form", "cat": "topics", "last_compiled": "2026-04-12T18:03Z"}, ...],
  "duplicate_suspects": ["arjun-gaur ~ arjun-gaur-clean", "sahil-sharma ~ sahil-sharma2", ...],
  "note": "Call find_similar_pages(slug) before creating a new page."
}
```
This kills the "I don't know what exists → I'll just create a new page" loop that produced the 11 duplicate pairs.

### 1.3 `stamp_page_compiled_at(file_path: str) -> dict[str, str]`
`compiler.py:93-128`. Sets `last_compiled`, `updated_by`, increments `update_count`.

| Criterion | Grade | Notes |
|---|---|---|
| Structured/useful | A | Returns all three fields the agent just wrote — the "echo the next-decision state" pattern. |
| Error surfacing | B- | Mixes string and bool: `{"ok": "false", "error": ...}` uses `"false"` as string, not bool. Minor but surprising. |
| Validates | C | Does not check the file is actually a wiki page or that frontmatter parsed — happily stamps a corrupt file, contributing to the double-`last_compiled` pathology (`audit-synthesis` §6). |
| Echoes state | A | Path, timestamp, model, count all returned. |
| Right abstraction | A | One trip replaces read → mutate → write. |
| Context cost | A | Tiny payload. |

Improvements: (a) return `ok: bool` not `"true"`/`"false"` strings; (b) abort if frontmatter already contains `last_compiled` twice and return `{"ok": false, "error": "duplicate last_compiled key in frontmatter", "hint": "call repair_frontmatter"}`; (c) refuse to stamp if frontmatter is unparseable so `update_wiki_index`'s `has_real_fields` heuristic isn't load-bearing (`compiler.py:213`).

### 1.4 `mark_as_compiled(file_path: str) -> dict[str, str | int]`
`compiler.py:131-169`. Flips `compiled: true` on a raw email, scans remaining uncompiled count, returns `{ok, remaining_uncompiled, path}`.

| Criterion | Grade | Notes |
|---|---|---|
| Structured/useful | A | `remaining_uncompiled` is the perfect "what do I do next" signal — Anthropic "Returning meaningful context". |
| Error surfacing | B | Returns on file-not-found, but no hint if called on a wiki file by mistake. |
| Validates | D | **Never verifies the email was actually compiled** — i.e., that any wiki page was modified that cites this raw path. This is why hallucinated entity pages still get `compiled: true` on their sources. |
| Echoes state | A | |
| Right abstraction | A | |
| Context | A | |

Improvements — this is the trust boundary, tighten it:
```python
@tool
def mark_as_compiled(
    file_path: str,
    cited_by: list[str],  # MANDATORY: wiki paths that now cite this raw file
) -> dict:
    """Refuses to mark compiled unless at least one wiki page's `sources:`
    now contains this raw path. Returns counts + a preview of the citing pages."""
```
Return example:
```json
{
  "ok": true,
  "remaining_uncompiled": 127,
  "citations_verified": 3,
  "citing_pages": ["wiki/topics/foo.md", "wiki/entities/bar.md", "wiki/systems/baz.md"],
  "warning": null
}
```
If `cited_by == []`, reject with `{"ok": false, "error": "refusing to mark compiled with no citations", "hint": "create or update at least one wiki page before marking"}`. This single change would have caught the Vikram Varshney hallucination case (entity page cites source, source marked compiled, nobody noticed the quote wasn't in the source body).

### 1.5 `update_wiki_index(wiki_dir: str = "wiki") -> str`
`compiler.py:172-255`. Regenerates `wiki/index.md` AND silently mutates N pages by auto-stamping `last_compiled`.

| Criterion | Grade | Notes |
|---|---|---|
| Structured/useful | C | Returns a single string: `"updated index: 461 pages ... auto-stamped 0 pages"`. Fine as log, not machine-readable. |
| Error surfacing | D | Silently `continue`s on YAML error (`compiler.py:228`); agent has no signal that some pages were skipped. |
| Validates | D | Does **two** side-effects (index write + page mutations) in one call. The `has_real_fields` heuristic at `compiler.py:213` exists specifically because this tool corrupted pages before. That's a design smell — a tool should not need a heuristic to avoid destroying files. |
| Echoes state | C | No list of *which* pages were stamped / skipped. |
| Right abstraction | F | Two tools in a trenchcoat. Split: `rebuild_index()` (pure read) and `bulk_stamp_missing_timestamps()` (explicit, opt-in side-effect). |
| Context | A | Small return. |

Also: `compile_all.py:209` already re-runs this after every batch outside the agent. The agent does not need this tool at all — remove from the registered tool list (`compiler.py:380`). Every registered tool burns a line of tool-description context on every turn (`context-engineering` post: "bloated tool sets that cover too much functionality").

### 1.6 `append_to_log(entry: str, wiki_dir: str = "wiki") -> str`
`compiler.py:258-282`. Appends `| timestamp | entry |` to `wiki/log.md`.

| Criterion | Grade | Notes |
|---|---|---|
| Structured/useful | B | String confirmation, fine. |
| Error surfacing | B | |
| Validates | D | No length cap — agent has appended 1 KB wall-of-text entries. No schema for what `entry` should contain. |
| Echoes state | D | Does not return number of entries in log or recent entries. Missed chance to remind the agent what was just done last. |
| Right abstraction | A | |
| Context | A | |

Improvements:
```python
def append_to_log(event_type: Literal["compile","skip","conflict","supersede","error"],
                  raw_path: str, pages_affected: list[str],
                  note: str = "", wiki_dir: str = "wiki") -> dict:
    """Enforces schema; returns {"ok": true, "entries_today": 47, "last_3": [...]}."""
```
Enforcing `event_type` makes the log greppable for later dashboards (Langfuse integration §4).

## 2. Deep Agents built-in filesystem tools (inherited, not declared by us)

These come from `deepagents.middleware.filesystem` when `create_deep_agent` is called with a `FilesystemBackend`. The agent sees them as first-class tools. We have **zero custom descriptions or wrappers** for them (`compiler.py:373-385`). All feedback below is against Anthropic's criteria as applied to *our* usage.

### 2.1 `read_file(file_path, offset=None, limit=None)`
Source: `.venv/.../deepagents/middleware/filesystem.py:709-806`. Returns file content with truncation-on-too-long.

Grade: **B for Deep Agents default, D for our wrapped usage.**

The problem is not the tool — it's that **a 307-entity wiki with 2-9 KB frontmatter blobs means every `read_file` on an entity page burns 2-9 KB of output tokens on frontmatter the agent cannot change and does not need**. The agent has no way to say "give me the body only".

Improvements — add a compiler-level wrapper:
```python
@tool
def read_wiki_page(slug: str, category: str | None = None,
                   sections: Literal["all","body","frontmatter","summary"] = "body",
                   section_names: list[str] | None = None) -> dict:
    """Returns only the requested portion. `summary` returns first 500 tokens of body + frontmatter minus `sources:`."""
```
Response example:
```json
{
  "path": "wiki/entities/himanshu-jain01.md",
  "title": "Himanshu Jain",
  "status": "current",
  "source_count": 50,
  "last_compiled": "2026-04-12T18:03Z",
  "body_tokens_approx": 680,
  "body_excerpt": "Himanshu is a product manager on the BuyerMy...",
  "h2_sections": ["Overview", "Recent Work", "Related"],
  "requested_section": "body",
  "content": "<body here>"
}
```
Adding this would cut per-email entity-page read cost by ~70-80% (the `sources:` block dominates). The agent should never have to read 50 raw filenames to add one sentence about someone.

### 2.2 `write_file(file_path, content)`
`deepagents/middleware/filesystem.py:811-856`. Writes; returns `"Updated file {path}"`.

Grade: **D against Anthropic playbook.**
- Response is a one-string `"Updated file <path>"`. Anthropic's "helpful error response" principle — but *also* "helpful success response" — is violated. The agent does not learn the resulting page size, whether frontmatter parsed, or whether any wikilinks are broken.
- No "read before write" enforcement — per the audit synthesis, this is how the 18 broken pages from 2026-04-13T02:05Z happened (agent overwrote without reading).
- No duplicate-file prevention. Agent can write `arjun-gaur-clean.md` next to `arjun-gaur.md` with zero friction (audit synthesis §1).

Improvements — wrap it at the compiler layer:
```python
@tool
def write_wiki_page(slug: str, category: Literal["topics","entities","systems","policies","timelines","conflicts"],
                    frontmatter: dict, body: str,
                    expected_prior_hash: str | None = None) -> dict:
    """Validates: category matches page_type; no suffix-dupe slugs (-new, -v2, -clean,
    trailing-digits); all [[wikilinks]] in body resolve; frontmatter YAML is well-formed;
    `sources:` are unique; no duplicate H2 sections. If expected_prior_hash is supplied
    and the file on disk does not match, rejects as stale (read-then-write enforcement)."""
```
Response example on success:
```json
{
  "ok": true,
  "path": "wiki/entities/arjun-gaur.md",
  "size_bytes": 3420,
  "wikilinks_resolved": 8,
  "wikilinks_broken": [],
  "new_page": false,
  "page_hash": "sha256:abcd..."
}
```
Response on refusal:
```json
{
  "ok": false,
  "error": "duplicate_suspect",
  "detail": "Slug 'arjun-gaur-clean' is similar to existing page 'arjun-gaur' (Levenshtein=6, suffix='-clean').",
  "hint": "Call merge_into('arjun-gaur-clean', into='arjun-gaur') or pick a distinct slug. NEVER use -clean / -new / -v2 / numeric suffixes — see prompt rule."
}
```
This is the single biggest win for the duplicate-page pain.

### 2.3 `edit_file(file_path, old_string, new_string, replace_all=False)`
`.venv/.../deepagents/middleware/filesystem.py:858-909`. Returns `"Successfully replaced N instance(s) of the string in '<path>'"`.

Grade: **C in isolation, F for our observed usage.**

`edit-tool-research-20260413T031839Z.md` already confirmed the tool itself is surgical — the problem is the agent pastes near-full-file `old_string`/`new_string` pairs on bloated pages. The tool does not resist this.

Improvements (some already proposed in that research doc; re-stating here with Anthropic grounding):
- Reject `len(old_string) + len(new_string) > 4096` with `{"ok": false, "error": "payload too large", "hint": "use execute sed for mechanical edits; for targeted hunks, pass ≤50 lines"}` — Anthropic "helpful error response" pattern.
- After-edit response returns `{"ok": true, "path": ..., "occurrences": 1, "new_size_bytes": 2134, "diff_summary": "-3 +2 lines", "frontmatter_parses": true, "wikilinks_broken_new": []}`. The frontmatter-parses check alone would kill the YAML-duplicate-key pathology.
- On `replace_all=True`, require `min_expected_occurrences` and `max_expected_occurrences` as guard rails.

### 2.4 `ls(path)`, `glob(pattern, path)`, `grep(pattern, path, glob)`
Built-ins, grades: **A, A, A-.** Already good. Only tweak: our system prompt pays the `ls` tool description (`deepagents/middleware/filesystem.py` ~150 tokens) on every turn for a tool that largely duplicates `list_wiki_pages`. Pick one.

### Minimal addition (ship this sprint)

**A. Enable Langfuse for real.** It's already a dependency (`pyproject.toml:21`). Just require the keys and fail loud if they're missing in non-dry-run mode:

```python
# scripts/compile_all.py, before the batch loop
if not dry_run and not (settings.langfuse_public_key and settings.langfuse_secret_key):
    raise SystemExit("LANGFUSE_* not set. Required for per-tool tracing. Set LANGFUSE_ENABLED=false to skip.")
```

Langfuse's `CallbackHandler` already attaches to the LangGraph agent; with it on, every tool invocation becomes a `Span` with input/output/latency/token count. Nothing else needs building for per-tool visibility.

**B. Wrap every custom tool in a decorator that records structured events** so we don't depend solely on Langfuse's UI for grep-able local data:

```python
# src/compile/telemetry.py (new)
def traced_tool(fn):
    @wraps(fn)
    def inner(*args, **kw):
        start = time.perf_counter()
        in_size = sum(len(str(a)) for a in args) + sum(len(str(v)) for v in kw.values())
        try:
            res = fn(*args, **kw)
            ok = True
            err = None
        except Exception as e:
            ok = False
            err = str(e)
            raise
        finally:
            logger.info("tool_call",
                        tool=fn.__name__,
                        ok=ok, err=err,
                        in_bytes=in_size,
                        out_bytes=len(str(res)) if ok else 0,
                        latency_ms=int((time.perf_counter()-start)*1000))
        return res
    return inner
```
Stack it under the existing `@tool` decorator for every function in `src/compile/compiler.py:23-282`.

**C. Add a per-batch summary log sink.** After each batch in `scripts/compile_all.py:173-206`, aggregate the structlog JSON for tool_call events and emit one final line:
```json
{"event":"batch_summary","batch":4,"emails":5,"duration_s":142,
 "tool_counts":{"read_file":18,"edit_file":9,"write_file":3,"stamp_page_compiled_at":6,"mark_as_compiled":5,"list_wiki_pages":1},
 "total_in_bytes":185402,"total_out_bytes":42013,
 "errors":{"edit_file":1},
 "pages_created":2,"pages_modified":4,
 "cost_usd_delta":0.42}
```
This is the "agent time spent" heatmap — eyeballing `tool_counts` across batches will immediately show `edit_file` looping on bloated pages vs. healthy `read_file:write_file ≈ 2:1` ratios.

Do **not** roll our own OpenTelemetry yet. Langfuse covers the LLM-side; structlog covers the tool-side; one ingestion is cheaper than two.

## 5. Framework-level guardrails (not per-tool)

These are the policies `deepagents` lets us hook via `middleware` / `patch_tool_calls` / `permissions`. See `.venv/.../deepagents/middleware/`.

### G1. Reject oversized `edit_file` payloads
Per `edit-tool-research-20260413T031839Z.md:67-70`. Hook `patch_tool_calls`: if `len(old_string) + len(new_string) > 4096` chars, replace the call with a tool-error response steering to `execute` (`sed`) or a smaller hunk. No retry loop needed — the model will adapt next turn.

### G2. Read-before-write enforcement
Track `files_read_this_turn` in agent state. In `write_file` / `edit_file` middleware, if `path not in files_read_this_turn` AND the file exists on disk, reject: `{"ok": false, "error": "write_without_read", "hint": "call read_file first to see current contents"}`. Identical to how Claude Code's text_editor enforces this.

### G3. Mandatory dry-run before destructive writes
For `write_wiki_page` and `merge_pages` (§3 items 4, 5), require `dry_run=True` on first call; return the diff; only on a second call with `confirm_hash` matching the dry-run can the write proceed. Two tool calls but surgical.

### G4. Page-size circuit breaker
If the target file is >20 KB, the agent's next tool call on that file is rerouted to a sub-agent with a focused prompt ("this page is oversized, your only job is to compact it: dedupe `sources:`, keep the 10 most recent, write a summary body"). Deep Agents already supports sub-agents via `deepagents.middleware.subagents` (`subagents.py`). This is exactly the "sub-agent architectures" pattern Anthropic describes: "specialized sub-agents can handle focused tasks with clean context windows."

### G5. Tool-budget cap per batch
Track per-batch tool call count. Hard-fail at 60 tool calls per batch (current `recursion_limit=150` translates to ~50-60 real tool calls after LLM turns). Log which tool exceeded the limit. This forces us to see infinite loops instead of silently burning budget.

### G6. Category invariant at write time
G1-G5 are generic; this one is wiki-specific. In the pre-write hook, reject any `wiki/entities/{slug}.md` whose body does not contain an `@` email pattern or a "first last" match to the slug; reject any `wiki/systems/{slug}.md` whose slug looks human ("first-last" pattern with no product suffix). Returns a hint with examples. Kills the 9 miscategorized pages found in the audit (synthesis §3) at creation time rather than letting them accumulate.

### G7. Namespace-tightened tool list
Remove `update_wiki_index` from the agent's toolbox (`compiler.py:380`) — it's invoked out-of-band by `compile_all.py:209`. Removing it also eliminates the "auto-stamp silently mutates pages" foot-gun. Net: one fewer tool description in the system prompt (Anthropic: "curating a minimal viable set of tools").
